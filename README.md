# 🔗 TrustChain: Multi-Agent Verification Environment

**GitHub Repository:** [https://github.com/HariniPrithiyangara/Trustchain-openenv](https://github.com/HariniPrithiyangara/Trustchain-openenv)  
**Hugging Face Space:** [https://huggingface.co/spaces/HariniPrithiyangara/trustchain-env](https://huggingface.co/spaces/HariniPrithiyangara/trustchain-env)

The #1 open problem in multi-agent AI is **trust and propagation**. When agents blindly trust outputs from other agents, a single hallucinated piece of information silently corrupts the entire pipeline, causing compounding errors across agentic workflows. 

TrustChain is a rigorous, real-world OpenEnv reinforcement learning environment that trains and evaluates agents to act as a **Verification Layer** in multi-agent pipelines. The agent operates like a fact-checker in a newsroom or an LLM safety auditor, receiving claims generated by a simulated "Reporter Agent".

---

## 🚀 Environment Overview

The agent must parse incoming claims alongside optional context, and confidently decide whether to **accept**, **reject**, or **cross-reference/request context** on ambiguous tasks.

This maps directly to real-world use cases:
- Content moderation pipelines
- Automated fact-checking systems
- Agentic AI safety layers
- LLM output auditing in multi-tier RAG architectures

## ♟️ Tasks & Difficulty Curriculum

TrustChain provides three distinct learning environments:

1. **Easy:** Fundamental ground-truth verifications. Claims are obviously true or demonstrably false. The agent must make a direct `pass_forward` or `flag_hallucination` judgment.
2. **Medium:** Nuanced reporting items. Claims require the agent to effectively use the `request_source` action before passing or flagging.
3. **Hard:** Subtle hallucinations woven into otherwise factual context. Designed to trick careless validation. The agent must thoroughly `cross_reference` context before flagging deceptive claims.

## 🛠️ State and Actions

**Observations (Type: Pydantic Model)**
- `claim_id (str)`: The unique identifier.
- `claim (str)`: The incoming claim to be verified.
- `sender (str)`: The AI reporter that generated the original text.
- `context (Optional[str])`: Additional context unlocked via agent investigation.
- `difficulty (str)`: The difficulty level of the task stack.
- `claims_remaining (int)`: How many validations are left in this episode.

**Actions (Type: Pydantic Model)**
- `action_type (Literal)`: The validation behavior mapping to exactly one of the following: `"verify_claim", "cross_reference", "flag_hallucination", "pass_forward", "request_source"`
- `claim_id (str)`: Target claim identifier.

## 📈 Reward Shaping

The environment utilizes a dense, proportional reward function:
- **True Positives / Accurate Action:** `+1.0` When an agent passes a factual claim or flags a hallucinated claim successfully.
- **Investigation Signals:** `+0.1` For appropriately requesting a source or cross-referencing an ambiguous claim. 
- **Incorrect Action Penalty:** `-0.5` Immediately applied for false-positive flagging or passing a hallucination forward.

Scores are constrained to `[0.0, 1.0]` across episodes.

## ⚙️ Quickstart

### Prerequisites
- Python 3.10+
- `openenv-core`
- `openai`

### Initializing Locally
```bash
# Provide inference variables
export API_BASE_URL="https://router.huggingface.co/v1"
export MODEL_NAME="meta-llama/Llama-3.3-70B-Instruct"
export HF_TOKEN="your-hf-token"

# Install package
pip install -e .

# Run inference script conforming to Hackathon standard
python inference.py
```

### Docker Verification Workflow
```bash
docker build -t trustchain-openenv .
docker run -p 8000:8000 trustchain-openenv
```

### OpenEnv CLI Tooling Validation
```bash
openenv validate
```
